❑ 데이터 및 모델 관리(Data and model management)
데이터 및 모델 관리는 Mlops 라이프 사이클의 중심적으로 교차되는 기능으로 ML 아티팩트를 관리하여 감사(auditability), 추적성(traceability) 및 규정 준수(compliance)를 지원하고 ML 자산의 공유성, 재사용성, 탐색성을 향상할 수 있다.

1. 데이터 세트 및 피처 관리 (Dataset and feature management)
데이터 과학 및 머신러닝의 주요 과제 중 하나는 모델 학습을 위한 고품질의 데이터를 생성하고 유지 및 재사용하는 것이다. 데이터 과학자는 탐색적 데이터 분석, 데이터 준비 그리고 데이터 변환에 상당한 노력과 시간이 소요된다. ML 모델을 개발하는 단계에서 유사한 ML 서비스에서 동일한 데이터 세트를 준비하지만, 데이터 과학자간에 데이터 세트를 공유하고 재사용할 수 있는 방법이 없다. 이러한 상황은 데이터 세트를 재개발하는 시간을 낭비할뿐만 아니라 동일한 데이터로 관리되는 엔티티의 일관되지 않은 정의로 데이터 품질의 저하로 이어질 수 있다. 또한 모델 예측 서빙 중에 제공되는 데이터와 모델 학습에서 사용한 데이터가 불일치하는 문제가 발생하며, 이를 학습–서빙 왜곡이라고 한다. 모델 학습과 모델 서빙 중에 다른 형태 또는 다른 데이터 소스에서 데이터가 추출되기 때문에 발생할 수 있으며, 운영 환경에서 모델 성능에 영향을 미치게 된다. 데이터 세트 및 피처 관리에서는 ML 모델에 사용되는 피처와 데이터 세트를 통합적으로 관리하는 저장소에 제공하여 재사용성을 높이고 학습–서빙 왜곡의 문제를 완화하는 데 도움이 된다.

2. 모델 관리(Model management)
데이터 과학자가 개발한 모델의 수가 증가하면서 모든 모델을 수동으로 추적하여 관리하기 어려워진다. ML 모델에 노출될 수 있는 위험을 관리하고, 모델의 책임을 구현하여 규정을 준수하기 위해서는 제어가 필요하다. 모델 관리는 MLOps의 중심에 있는 모든 영역에서 공통적으로 관리되어야 하는 프로세스로, ML 메타데이터의 추적과 모델 거버넌스를 모두 포함한다. ML 라이프사이클을 통해서 모델을 관리하고 있다면 다음을 보장할 수 있다.
• 모델 학습 및 평가에 수집되고 사용되는 데이터는 정확하고 편견이 없으며 개인 정보 위반없이 적절하게 사용된다.
• 모델은 효율성 및 품질을 측정하고 공정성 지표에 대한 평가 및 검증으로 운영 환경에 배포된다.
• 필요에 따라서 모델은 예측 결과에 대한 해석이 가능하며 설명할 수 있다.
• 운영 서비스에 배포된 모델의 성능은 지속적인 평가로 모니터링되며 모델의 성능 메트릭이 추적되고 모니터링된다.
• 모델 학습 또는 예측 서빙의 잠재적인 문제는 추적 가능하고, 디버깅이 가능하며 재현 가능하다.

3. ML 메타 데이터 추적(ML metadata tracking)
각 프로세스에 의해서 생성된 아티팩트는 일반적으로 프로세스 실행에 대한 정보와 함께 ML 아티팩트 저장소에 자동으로 저장된다. ML 메타 데이터에는 파이프라인 실행 ID, 트리거, 프로세스 유형, 단계, 시작 및 종료 시간, 상태, 환경 구성 및 매개 변수값 등이 포함될 수 있다. 모델의 아티팩트의 예시로는 데이터 세트와 스키마, 통계 데이터, 하이퍼 파라미터, ML모델 그리고 모델의 평가지표 등이 포함된다. ML 메타 데이터 추적을 통해서 데이터 과학자와 ML 엔지니어는 계보 추적과 재현성을 위해서 실험의 매개 변수와 파이프라인의 설정을 추적할 수 있다. 기존 ML 모델과 아티팩트를 쉽게 검색하고 다운로드 받을 수 있으며, 다양한 모델 실험과 ML 파이프라인의 실행으로 메타 데이터와 아티팩트를 분석하고 비교할 수 있다. 이를 통해서 데이터 과학자와 ML 엔지니어는 ML 파이프라인의 동작을 이해하고 ML 문제를 디버깅하는 데 도움이 된다.

4. 모델 거버넌스(Model governance)
모델 거버넌스는 모델을 서비스에 배포하기 위해서 등록, 검토, 검증 및 승인에 대한 프로세스로 기업과 모델 규제의 요구사항 또는 특정 사용 사례에 따라 모델 거버넌스 프로세스가 다를 수 있다. 거버넌스 프로세스는 ML 모델이 운영에 배포될 준비가 되었는지를 결정하기 위해서 자동화 또는 반자동화 할 수 있으며, 운영에 배포된 모델의 성능에 대한 거버넌스를 위한 보고를 지원해야 한다. 리스크 관리자와 감사자에게 모델의 계보와 책임에 대한 명확한 정보를 제공해야 하며, 이 과정은 조직의 윤리 그리고 법적 책임에 따라 모델 거버넌스의 결정을 검토 할 수 있는 기능을 제공해야 한다.








모델 거버넌스는 ML 메타 데이터 및 모델 레지스트리의 정보를 사용하여 다음과 같은 작업을 수행 할 수 있다.
• 저장소 : 모델 속성을 추가 또는 업데이트하고 모델 버전 및 속성 변경을 추적합니다. 모델 레지스트리는 실험 및 지속적 학습 단계에서 많은 모델 버전을 저장할 수 있어 데이터 과학자가 중요한 모델을 쉽게 재현 할 수 있다.


Model governance is about registering, reviewing, validating, and approving models for deployment.
모델 거버넌스는 배포를위한 모델 등록, 검토, 검증 및 승인에 관한 것입니다.
Depending on the organization, on the regulatory requirements of the model, and on the particular use case, the process of model governance can differ.
조직, 모델의 규제 요구 사항 및 특정 사용 사례에 따라 모델 거버넌스 프로세스가 다를 수 있습니다.
The process can be automated, semi–automated, or fully automated (with multiple release criteria in all cases) to determine whether ML models are ready to go to production.
이 프로세스는 ML 모델이 생산에 갈 준비가되었는지 여부를 결정하기 위해 자동화, 반자동 또는 완전 자동화 (모든 경우에 여러 릴리스 기준) 할 수 있습니다.
In addition, model governance should support reporting on the performance of deployed models.
또한 모델 거버넌스는 배포 된 모델의 성능에 대한 보고를 지원해야합니다.


Figure 14 shows the tasks that are involved in model governance.
그림 14는 모델 거버넌스와 관련된 작업을 보여줍니다.


Model governance can use information in the ML metadata and the model registry to do the following tasks:
모델 거버넌스는 ML 메타 데이터 및 모델 레지스트리의 정보를 사용하여 다음 작업을 수행 할 수 있습니다.


• Store: Add or update model properties and track model versions and property changes. The model registry can store many model versions from the experimentation and continuous training phases, which lets data scientists easily reproduce significant models.
• 저장 : 모델 속성을 추가 또는 업데이트하고 모델 버전 및 속성 변경을 추적합니다. 모델 레지스트리는 실험 및 연속 교육 단계에서 많은 모델 버전을 저장할 수있어 데이터 과학자가 중요한 모델을 쉽게 재현 할 수 있습니다.
• Evaluate: Compare a new challenger model to the champion model by looking not only at evaluation metrics (accuracy, precision, recall, specificity, and so on) but also at business KPIs that are collected through online experimentation.
• 평가 : 평가 메트릭 (정확도, 정밀, 리콜, 특이성 등)뿐만 아니라 온라인 실험을 통해 수집 된 비즈니스 KPI를 통해 새로운 도전자 모델을 챔피언 모델과 비교하십시오.
Additionally, model owners need to be able to understand and explain the model predictions—for example, by using feature attribution methods. This ensures the quality of the model that is deployed in production.
또한 모델 소유자는 기능 속성 방법을 사용하여 모델 예측을 이해하고 설명 할 수 있어야합니다.이를 통해 생산에 배치 된 모델의 품질을 보장합니다.
• Check: Review, request changes, and approve the model to help control for risks, such as business, financial, legal, security, privacy, reputational, and ethical concerns.
• 확인 : 비즈니스, 재무, 법률, 보안, 개인 정보 보호, 평판 및 윤리적 문제와 같은 위험을 제어 할 수 있도록 모델을 검토하고 요청하고 승인합니다.
• Release: Invoke the model deployment process to go live. This controls the type of the model release (for example, canary or blue–green) and the traffic split that is directed to it.
• 릴리스 : 모델 배포 프로세스를 호출하여 생중계됩니다.이는 모델 릴리스 유형 (예 : 카나리아 또는 청록색)과 그에 대한 트래픽 분할을 제어합니다.
• Report: Aggregate, visualize, and highlight model performance metrics that are collected from the continuous evaluation process. This ensures the quality of the model in production.
• 보고서 : 지속적인 평가 프로세스에서 수집 된 모델 성능 메트릭을 집계, 시각화 및 강조합니다.이것은 생산에서 모델의 품질을 보장합니다.


Explainability is particularly important in the case of decision automation.
결정 자동화의 경우 설명 가능성이 특히 중요합니다.
The governance process should provide to risk managers and auditors a clear view of lineage and accountability.
The process should also provide them the ability to review decisions in accordance with an organization’s ethical and legal responsibilities.
거버넌스 프로세스는 리스크 관리자와 감사자에게 모델의 계보와 책임에 대한 명확한 견해를 제공해야합니다.이 과정은 또한 조직의 윤리적, 법적 책임에 따라 결정을 검토 할 수있는 능력을 제공해야합니다.

1) Feature management
1) 기능 관리
Features are attributes of business entities that are cleansed and prepared based on standard business rules—aggregations, derivations, flags, and so on.
특징은 표준 비즈니스 규칙을 기반으로 정리하고 준비된 비즈니스 엔티티의 속성과 같은 비즈니스 엔티티의 속성입니다.
Examples of entities include product, customer, location, and promotion.
엔티티의 예로는 제품, 고객, 위치 및 프로모션이 있습니다.
You can manage your data entities in a centralized repository to standardize their definition, storage, and access for training and serving.
중앙 저장소에서 데이터 엔티티를 관리하여 교육 및 서빙을위한 정의, 저장 및 액세스를 표준화 할 수 있습니다.
A feature repository helps data scientists and researchers do the following:
기능 저장소는 데이터 과학자와 연구원이 다음을 수행하는 데 도움이됩니다.


• Discover and reuse available feature sets for their entities instead of re–creating the entities in order to create their own datasets.
• 자신의 데이터 세트를 만들기 위해 엔티티를 다시 만들지 않고 엔티티에 사용 가능한 기능 세트를 발견하고 재사용합니다.
• Establish a central definition of features.
• 기능의 중심 정의를 설정합니다.
• Avoid training–serving skew by using the feature repository as the data source for experimentation, continuous training, and online serving.
• 실험, 지속적인 교육 및 온라인 서빙의 데이터 소스로 기능 저장소를 사용하여 훈련–서빙 왜곡을 피하십시오.
• Serve up–to–date feature values from the feature repository.
• 기능 저장소에서 최신 기능 값을 제공합니다.
• Provide a way of defining and sharing new entities and features.
• 새로운 엔티티 및 기능을 정의하고 공유하는 방법을 제공합니다.
• Improve collaboration between data science and research teams by sharing features.
• 기능을 공유하여 데이터 과학 및 연구 팀 간의 협업을 향상시킵니다.


In batch ETL systems, the training pipeline can retrieve the features as a batch for the training task.
배치 ETL 시스템에서 교육 파이프 라인은 기능을 교육 작업의 배치로 검색 할 수 있습니다.
For online serving, the serving engine can fetch the feature values that are related to the requested entity. Updates to the feature repository can be ingested from the batch ETL or streaming systems.
온라인 서빙의 경우 서빙 엔진은 요청 된 엔티티와 관련된 기능 값을 가져올 수 있습니다.기능 저장소에 대한 업데이트는 배치 ETL 또는 스트리밍 시스템에서 섭취 할 수 있습니다.
In addition to those updates, the monitoring service can update statistics and metrics for these features.
이러한 업데이트 외에도 모니터링 서비스는 이러한 기능에 대한 통계 및 메트릭을 업데이트 할 수 있습니다.


2) Dataset management
2) 데이터 세트 관리
Features can be used in many datasets for multiple ML tasks and use cases, while a dataset is used for a particular ML task or use case.
기능은 여러 ML 작업 및 사용 사례에 대한 많은 데이터 세트에서 사용될 수 있으며 데이터 세트는 특정 ML 작업 또는 사용 사례에 사용됩니다.
More precisely, feature repositories typically don’t include labeled data instances (instances with predictable targets).
보다 정확하게는 기능 저장소에는 일반적으로 레이블이 붙은 데이터 인스턴스 (예측 가능한 대상이있는 인스턴스)가 포함되지 않습니다.
Instead, they include reusable feature values of various entities.
대신, 여기에는 다양한 엔티티의 재사용 가능한 기능 값이 포함됩니다.
The features of different entities can be combined and joined with other transactional data that contains labels in order to create a dataset.
다른 엔티티의 기능은 데이터 세트를 만들기 위해 레이블이 포함 된 다른 트랜잭션 데이터와 결합하여 결합 할 수 있습니다.


For example, the feature repository might contain a customer entity that includes features that describe customer demographics, purchase behavior, social media interactions, sentiment scores, third–party financial flags, and so on.
예를 들어, 기능 저장소에는 고객 인구 통계, 구매 행동, 소셜 미디어 상호 작용, 정서 점수, 타사 금융 플래그 등을 설명하는 기능이 포함 된 고객 엔티티가 포함될 수 있습니다.
The customer entity can be used in several tasks, such as churn prediction, click–through rate prediction, customer lifetime value estimation, customer segmentation, and recommendations.
고객 엔티티는 이탈 예측, 클릭률 예측, 고객 수명 가치 추정, 고객 세분화 및 권장 사항과 같은 여러 작업에서 사용할 수 있습니다.
Each task has its own dataset that contains the customer features and other features from the entities that are relevant to the task.
각 작업에는 작업과 관련된 엔티티의 고객 기능 및 기타 기능이 포함 된 자체 데이터 세트가 있습니다.
In addition, in case of supervised learning tasks, each dataset has its own labels.
또한 감독 학습 작업의 경우 각 데이터 세트에는 자체 라벨이 있습니다.


Dataset management helps with the following:
데이터 세트 관리는 다음과 같은 도움이됩니다.
• Maintaining scripts for creating datasets and splits so that datasets can be created in different environments (development, test, production, and so on).
• 데이터 세트 및 분할을 생성하기위한 스크립트를 유지하여 데이터 세트 (개발, 테스트, 생산 등)에서 데이터 세트를 생성 할 수 있습니다.
• Maintaining a single dataset definition and realization within the team to use in various model implementations and hyperparameters.
